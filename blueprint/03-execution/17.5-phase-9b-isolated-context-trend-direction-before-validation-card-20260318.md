## 4. 验证边界

本轮只允许改一个主变量：

`add context_trend_direction_before parent-context guard semantics to the validated baseline`

但 runtime 真正消费的字段名必须写诚实：

`current_context_trend_direction`

本轮显式禁止：

1. `context_trend_direction_before + wave_role`
2. `context_trend_direction_before + duration_percentile`
3. `context_trend_direction_before + reversal_state`
4. 任意 `mirror / conditioning / gene_score` 伴随进入
5. 直接把它写成组合 package 的一部分

## 5. 当前冻结口径

本卡当前不再只冻结角色，也把第一条实际候选规则写死。

当前被允许的角色只有：

1. `parent-context negative guard`
2. `directional admissibility check`

当前默认先验证的唯一窄规则是：

`block when current_context_trend_direction == DOWN`

这条规则在命名上应理解为：

`用 runtime snapshot 字段 current_context_trend_direction，去验证 ledger 侧 context_trend_direction_before 的 parent-context negative guard 语义。`

当前明确不允许的角色有：

1. `default hard gate`
2. `sizing overlay`
3. `多字段组合 companion`

## 7. 最终裁决

本轮正式 ruling 是：

`promote_context_trend_direction_negative_guard`

原因是：

1. `signals_count` 保持 `16` 不变
2. candidate 真实拦掉了 `6` 个 signals，其中 `5` 个是 baseline 的真实 filled entry
3. 被拦对象里有 `4` 个 `MAINSTREAM`，说明它不是 `wave_role` 的换名复跑
4. full-window `expected_value` 从 `-0.0135471571` 收窄到 `-0.0043857680`
5. `profit_factor / max_drawdown / reject_rate` 同时改善

一句话解释：

`这条 parent-context negative guard 没有像 wave_role 那样重压交易数量，而是更轻地切掉了父趋势向下的高风险 entry。`

即使本卡已经完成，也不代表：

1. `Gene` 已整体进入默认 runtime
2. `Phase 9` 整包已经 closeout
3. `17.6 / Phase 9C` 已自动打开
4. `duration_percentile + reversal_state + context_trend_direction_before` 现在可以自动组合
5. `context_trend_direction_before` 已经能直接扩成 default hard gate 或 sizing overlay

本卡只说明：

`context_trend_direction_before` 已完成 truthful isolated validation，并赢下 narrow parent-context negative guard 这一轮。`
