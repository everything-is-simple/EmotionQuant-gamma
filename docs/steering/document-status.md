# 文档状态语义

**版本**: `v0.01-v0.06 治理规则`  
**状态**: `Frozen`  
**封版日期**: `2026-03-07`  
**变更规则**: `仅允许勘误、链接修复与示例补充；若状态枚举或定义发生变化，必须先修订上游 baseline 与相关入口审计。`  
**上游文档**: `docs/design-v2/01-system/system-baseline.md`

## 冻结区与冲突处理

1. 本文档定义仓库文档状态字段的统一语义；各目录 README、治理文档、设计文档不得自行发明新状态。
2. 若文档状态语义与 `docs/design-v2/01-system/system-baseline.md` 或当前治理状态冲突，以 baseline 和 `docs/spec/common/records/development-status.md` 为准。
3. 状态字段用于说明“文档维护方式”，不是说明“功能已经实现”。
4. 版本证据、审计记录与状态变更留痕，统一归档到 `docs/spec/<version>/` 或 `docs/spec/common/records/`。

## 1. 适用范围

适用于以下文档头部的 `状态` 字段：

- `docs/design-v2/`
- `docs/steering/`
- `docs/spec/` 下承担治理或入口职责的 README / records

其他目录若新增头部状态字段，也必须复用本定义。

## 2. 状态枚举

| 状态 | 含义 | 允许变更 | 典型位置 |
|---|---|---|---|
| `Frozen` | 口径已冻结，只允许勘误、链接修复、说明性澄清 | 小修，不改执行语义 | `steering/`、冻结版设计文档、目录入口 |
| `Active` | 当前仍是维护中的权威口径，可在既定边界内持续修订 | 允许受控调整内容，但不能越过上游约束 | 算法级 SoT、当前状态文档 |
| `Draft` | 草案或待验证方案，尚未成为正式执行口径 | 允许继续收敛、删除、重构 | 方案草案、待证据确认的集成设计 |

## 3. 头部字段规则

统一字段顺序如下：

1. `版本`
2. `状态`
3. `封版日期`
4. `变更规则`
5. `上游文档`

其余字段如 `创建日期`、`最后更新`、`对应模块`、`理论来源`、`证据支持` 放在这 5 项之后。

## 4. 封版日期规则

| 状态 | `封版日期` 写法 | 说明 |
|---|---|---|
| `Frozen` | 具体日期，如 `2026-03-07` | 表示正式冻结时间 |
| `Active` | `不适用（Active SoT）` | 仍在维护，不宣称已封版 |
| `Draft` | `不适用（Draft）` | 仍在迭代，不宣称已封版 |

禁止出现以下混用：

- `Active` + 具体封版日期
- `Draft` + 具体封版日期
- `Frozen` + `不适用`

## 5. 使用约束

1. `Frozen` 不能用来表达“这个话题没人维护了”，只能表示“口径冻结”。
2. `Active` 不能用来表达“开发正在推进”，只能表示“文档仍是当前维护中的权威口径”。
3. `Draft` 不能被 README、AGENTS、路线图等入口文档当作正式 SoT 引用。
4. 若一份文档从 `Draft` 转为 `Frozen` 或 `Active`，必须同步修订相关 README、入口审计或 development-status。

## 6. 维护动作

发生以下情况时，必须检查本规则是否需要同步：

- 新增带头部元数据的治理/设计文档
- 既有文档从草案转为正式口径
- README 或 AGENTS 对文档状态含义作出解释
- `preflight`、文档 gate 或入口审计脚本新增状态相关检查
