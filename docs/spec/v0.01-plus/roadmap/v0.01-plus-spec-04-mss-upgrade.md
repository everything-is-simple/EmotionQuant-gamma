# v0.01-plus Spec-04 MSS-upgrade

**版本**: `v0.01-plus 主开发线`
**状态**: `Active`
**封版日期**: `不适用（Active SoT）`
**变更规则**: `允许在不改变 v0.01 Frozen 历史基线的前提下，对 MSS 周期层、趋势层、仓位建议层与执行层对接方式做受控修订。`
**上游文档**: `docs/spec/v0.01-plus/roadmap/v0.01-plus-roadmap.md`, `docs/design-v2/03-algorithms/core-algorithms/mss-algorithm.md`
**对应代码**: `src/selector/mss.py`, `src/broker/risk.py`

---

## 1. 目标

当前主线中的 `MSS` 已经完成“从前置 gate 收口到执行层风控”的第一步，但它仍然只是 `MSS-lite`。

`Spec-04` 的目标不是把 `MSS` 重新塞回排序层，而是恢复原始 `MSS-full` 里真正有价值、且仍然适合当前主线的四层能力：

1. 周期层
2. 趋势层
3. 仓位建议层
4. 风险状态机

---

## 2. 当前问题

当前 `MSS-lite` 只稳定输出：

- `score`
- `signal`
- 风险倍率映射

它没有完整回答以下问题：

1. 当前市场处于哪一个情绪周期阶段。
2. 当前阶段是在上升、下降还是横盘。
3. 当前阶段已经持续了多久。
4. 风险倍率和原始仓位建议之间如何一一对应。

这会带来两个直接问题：

1. 理论口径和运行时口径脱节。
2. `Broker / Risk` 虽然能调仓位，但解释层仍然过薄。

---

## 3. 恢复范围

### 3.1 周期层

恢复原始七阶段情绪周期：

1. `EMERGENCE`
2. `FERMENTATION`
3. `ACCELERATION`
4. `DIVERGENCE`
5. `CLIMAX`
6. `DIFFUSION`
7. `RECESSION`

要求：

- 周期判定必须基于 `temperature + trend`
- 周期判定必须可回放
- 周期输出必须落库

### 3.2 趋势层

恢复趋势方向字段：

- `UP`
- `DOWN`
- `SIDEWAYS`

要求：

- 趋势判定只依赖 as-of 可得温度序列
- 当前起步版允许使用最近 `3` 个交易日温度严格递增/递减规则
- 后续可扩为更稳健的平滑趋势规则

### 3.3 仓位建议层

恢复原始仓位建议输出，例如：

- `80%-100%`
- `60%-80%`
- `50%-70%`
- `40%-60%`
- `30%-50%`
- `20%-40%`
- `0%-20%`

要求：

- 仓位建议来源于 `cycle + trend`
- `Broker / Risk` 的风险倍率应能追溯到该仓位建议
- 不允许只保留倍率、丢掉文字仓位区间

### 3.4 风险状态机

恢复更正式的状态机语义：

- `phase`
- `phase_trend`
- `phase_days`
- `position_advice`
- `risk_regime`

其中：

- `risk_regime` 用于执行层
- `phase / trend / phase_days / position_advice` 用于解释层与审计层

---

## 4. 数据模型扩展

在当前 `l3_mss_daily` 之上，下一步补齐以下字段：

- `phase`
- `phase_trend`
- `phase_days`
- `position_advice`
- `risk_regime`

说明：

1. `score / signal` 继续保留，作为兼容字段。
2. 新字段先进入 `l3_mss_daily`，不需要等待契约大改版。
3. `Broker / Risk` 优先消费 `risk_regime`，其余字段主要供报告与证据层使用。

---

## 5. 分批实施

### Phase A：周期层补回

范围：

- `phase`
- `phase_trend`
- `phase_days`
- `position_advice`

目标：

- 先把 `MSS-full` 的解释骨架补回来
- 不立即改变当前风险倍率逻辑

### Phase B：风险状态机补回

范围：

- `risk_regime`
- `position_advice -> risk overlay` 映射表

目标：

- 让 `Broker / Risk` 不再只看三态
- 让倍率和周期语义一一对应

### Phase C：长窗口校准

范围：

- 周期阈值
- 趋势规则
- 仓位区间到倍率映射

目标：

- 用长窗口证据验证：
  - 是否改善 `MDD`
  - 是否改善收益结构
  - 是否降低高温区回撤暴露

---

## 6. 设计约束

1. `MSS` 不回到 `Selector` 前置 gate。
2. `MSS` 不进入个股横截面 `final_score`。
3. `MSS` 的主线职责仍然是市场级风险调节。
4. 新增周期字段必须服务于解释与执行，不得重新制造漏斗式交易决策。

---

## 7. 验收标准

### 7.1 文档验收

需要能明确回答：

1. 当前市场处于哪个周期。
2. 当前周期方向是什么。
3. 当前周期持续了几天。
4. 当前建议仓位是什么。
5. 当前执行倍率为什么是这个值。

### 7.2 数据验收

`l3_mss_daily` 至少可追溯：

- `score`
- `signal`
- `phase`
- `phase_trend`
- `phase_days`
- `position_advice`
- `risk_regime`

### 7.3 执行验收

`Broker / Risk` 能基于 `risk_regime` 或其等价映射稳定复现：

- `max_positions`
- `risk_per_trade_pct`
- `max_position_pct`

---

## 8. 与路线图关系

`Spec-04` 属于 `v0.01-plus` 当前主开发线，不后推为纯 `v0.02` 话题。

原因：

1. 原始 `MSS` 的核心价值就在周期和仓位建议。
2. 如果当前主线只保留 `score + signal`，`MSS` 会继续停留在“阉割版可跑”状态。
3. 当前代码已经接入 `Broker / Risk`，现在补周期层的边际成本低于以后再补。
