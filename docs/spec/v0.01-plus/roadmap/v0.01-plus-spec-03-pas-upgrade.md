# v0.01-plus Spec-03：PAS 升级设计

- **版本**: `v0.01-plus Spec-03`
- **状态**: `Active`
- **日期**: `2026-03-08`
- **基线**: `docs/Strategy/PAS/`, `docs/design-v2/03-algorithms/core-algorithms/pas-algorithm.md`, `docs/design-v2/03-algorithms/core-algorithms/down-to-top-integration.md`
- **治理状态**: `docs/spec/common/records/development-status.md`

## 1. 定位

这份文档回答的不是“当前 PAS-trigger 怎么跑”，而是：

`如何把原始 PAS-full 里真正有价值的内容，分批恢复到当前 v0.01-plus 主线。`

当前决策已经明确：

1. 当前主线实现的只是 `PAS-trigger`
2. 这和 `docs/Strategy/PAS/` 里的原始 `PAS-full` 有明显差距
3. `PAS` 不该长期停留在“只有 BOF 触发器”的状态
4. 但恢复必须分批、可验证，不能一口气把完整理论体系硬塞进主线

---

## 2. 当前缺口

相对 `docs/Strategy/PAS/` 的原始定义，当前主线缺三大块：

1. **形态生态缺失**
   - 当前只有 `BOF`
   - 缺 `BPB / TST / PB / CPB`

2. **形态质量层缺失**
   - 当前只有 trigger
   - 没有对结构清晰度、量能确认、位置优势、失败风险做系统评分

3. **交易管理参考层缺失**
   - 当前 formal signal 不包含止损/目标位/风险收益比
   - 也没有失败形态处理

---

## 3. 总体策略

恢复 `PAS-full`，但不破坏当前主线分层：

`Selector 初选 -> PAS-trigger -> IRS 排序 -> MSS 控仓位 -> Broker 执行`

因此必须坚持：

1. `PAS` 可以扩，但不能重新吞掉 `IRS` 和 `MSS` 的职责
2. `PAS` 增强的是“个股形态层”，不是行业层和市场层
3. 任何恢复都必须沿着 `detector / registry / strategy` 当前骨架推进

---

## 4. 三批恢复计划

## 4.1 第一批：形态生态恢复

目标：

把原始 PAS 中最重要的几种形态逐步接回当前主线。

范围：

1. `BPB`
2. `TST`
3. `PB`
4. `CPB`

当前建议顺序：

1. `BPB`
2. `TST`
3. `PB`
4. `CPB`

原因：

- `BPB` 与当前 `BOF` 互补最强
- `TST / PB` 次之
- `CPB` 最复杂，放最后

交付物：

1. `src/strategy/pas_bpb.py`
2. `src/strategy/pas_tst.py`
3. `src/strategy/pas_pb.py`
4. `src/strategy/pas_cpb.py`
5. `registry.py` 扩展
6. 单形态回测与组合回测证据

验收方式：

- 一形态一 detector
- 一形态一组单测
- 一形态一轮独立回测
- 通过后才允许进入组合模式

---

## 4.2 第二批：形态质量评分恢复

目标：

让 `PAS` 不只回答“有没有触发”，还回答“这个触发质量怎么样”。

建议恢复的质量层因子：

1. `structure_clarity`
2. `volume_confirmation`
3. `position_advantage`
4. `failure_risk`

建议输出：

- `pattern_quality_score`
- `quality_breakdown_json`

当前边界：

- 这层先进入 `DTT sidecar`
- 不急着改 formal schema
- 是否进入 `final_score`，要通过单独消融决定

交付物：

1. `src/strategy/pattern_quality.py`
2. `l3_signal_rank_exp` 扩展字段
3. `BOF-only` 与 `BOF+quality` 消融证据

---

## 4.3 第三批：交易管理参考恢复

目标：

把原始 PAS 中交易管理最有价值、最可解释的部分，恢复为参考层，而不是直接侵入 Broker 内核。

建议恢复的内容：

1. `stop_price`
2. `target_price`
3. `risk_reward_ratio`
4. `failure_handling_tag`

当前约束：

- 这批先作为参考层输出，不直接改写当前 Broker 的正式风控核心
- 先解决“解释”和“证据”，再决定是否进入执行

交付物：

1. `pas-trade-management.md` 或并入 `pas-data-models.md`
2. sidecar 字段扩展
3. 失败形态案例与证据记录

---

## 5. 分期与版本关系

### 5.1 提前纳入 v0.01-plus 的内容

建议提前纳入 `v0.01-plus`：

1. `BPB`
2. `pattern_quality_score`
3. `stop / target / RR` 参考层

原因：

- 这三项最能明显缩小 `PAS-trigger` 与 `PAS-full` 的差距
- 又不会一下把系统复杂度拉爆

### 5.2 继续留在 v0.02+ 的内容

建议继续留在 `v0.02+`：

1. `TST / PB / CPB` 的完整生态
2. 失败形态反向信号系统
3. 更复杂的自适应参数

---

## 6. 与主线边界的硬约束

1. `PAS` 不得吞掉 `IRS` 的行业排序职责
2. `PAS` 不得吞掉 `MSS` 的市场级控仓位职责
3. `PAS` 可增强个股层，但不回退成“个股总分吞一切”的旧混合模型
4. `PAS` 的新增能力，优先走 detector 和 sidecar，不急着直接改 formal schema

---

## 7. 建议实现顺序

### Phase A

1. `BPB detector`
2. `BPB` 单形态证据
3. `BOF + BPB` 组合证据

### Phase B

1. `pattern_quality_score`
2. `quality` sidecar 字段
3. `BOF/BOF+BPB + quality` 消融

### Phase C

1. `stop / target / RR` 参考层
2. `failure_handling_tag`
3. 失败形态样本归档

---

## 8. 验收标准

### 8.1 设计验收

- [ ] `PAS-trigger` 与 `PAS-full` 边界写清楚
- [ ] 三批恢复内容各自独立定义
- [ ] 不与 `IRS / MSS / Broker` 职责冲突

### 8.2 证据验收

至少完成：

1. `BOF`
2. `BOF + BPB`
3. `BOF + quality`
4. `BOF + BPB + quality`

并比较：

- `trade_count`
- `EV`
- `PF`
- `MDD`
- `rank_diff_days`
- `execution_diff_days`

### 8.3 主线验收

只有当新增 `PAS` 能力：

- 不恶化当前主线稳定性
- 能解释收益变化来源
- 不破坏当前 DTT 分层

才允许把它并入主线默认配置。

---

## 9. 当前结论

当前结论已经明确：

1. 现在的 `PAS` 只是 `PAS-trigger`
2. 原始 `PAS-full` 的很多关键价值还没回到主线
3. 最值得先恢复的不是“全部理论”，而是三批最有实现价值的内容：
   - `BPB/TST/PB/CPB`
   - 形态质量评分
   - 止损/目标位/失败处理参考层

其中，第一批建议优先做：

- `BPB`
- `pattern_quality_score`
- `stop / target / RR`

这样最务实，也最接近原始 PAS 的真正价值。
