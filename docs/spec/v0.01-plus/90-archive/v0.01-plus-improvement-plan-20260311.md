## 2. v0.01-plus 的核心问题诊断

### 2.1 问题 1：MSS 动态缩容实现有致命缺陷

**现状**：
```
2026-02-06 / 2026-02-09 出现 over-cap carryover：
- holdings_before = 5
- effective_max_positions = 4
- 新信号（rank=1）被直接拒绝
- 没有延迟入场机制
```

**问题**：
- 允许持仓数超过动态上限
- 新机会被"满槽即拒"
- 7 笔被拒交易中有 5 笔是赚钱的（+3087 元）

**根因**：
- Broker 实现时没有考虑 over-cap 场景
- 缺少"先平后进"或"延迟入场"机制

### 2.2 问题 2：IRS 排序完全无效

**现状**：
```
所有对比实验中：
- rank_changed_count = 0
- IRS 没有改变任何股票的排名
```

**问题**：
- IRS 模块虽然运行了，但没有起作用
- 相当于白白增加了系统复杂度

**根因**（待验证）：
1. IRS score 计算有 bug
2. 权重设置为 0 或被覆盖
3. 排序逻辑没有正确集成到 Strategy

## 3. 改进方案

### 3.1 改进 1：修复 MSS over-cap carryover（优先级 P0）

#### 3.1.1 目标

解决持仓数超过动态上限时，新机会被拒绝的问题。

#### 3.1.2 具体方案

**方案 A：禁止 over-cap（激进）**

```python
# src/broker/broker.py

def process_buy_signal(self, signal: Signal, execute_date: str):
    """处理买入信号"""
    # 获取当前持仓数
    current_holdings = len(self.positions)
    
    # 获取动态上限
    effective_max_positions = self.risk.get_effective_max_positions(execute_date)
    
    # 新增：over-cap 检查
    if current_holdings >= effective_max_positions:
        # 找出最弱的持仓
        weakest_position = self._find_weakest_position()
        
        if weakest_position:
            # 先平掉最弱持仓，腾出空间
            self._close_position(weakest_position, execute_date, reason="MAKE_ROOM_FOR_NEW")
            logger.info(f"Over-cap detected, closed {weakest_position.code} to make room")
        else:
            # 如果没有可平仓位，拒绝新信号
            return self._reject_order(signal, "MAX_POSITIONS_REACHED")
    
    # 继续正常买入流程
    return self._create_buy_order(signal, execute_date)

def _find_weakest_position(self) -> Optional[Position]:
    """找出最弱的持仓（用于 over-cap 时腾空间）"""
    if not self.positions:
        return None
    
    # 按浮动盈亏排序，选择最差的
    sorted_positions = sorted(
        self.positions.values(),
        key=lambda p: p.unrealized_pnl
    )
    
    return sorted_positions[0]
```

**预期效果**：
- 不再出现 over-cap 拒单
- 动态调整持仓，保持在上限以内
- 风险：可能过早平掉潜力股

**方案 B：延迟入场队列（保守）**

```python
# src/broker/broker.py

class Broker:
    def __init__(self):
        self.pending_signals = []  # 新增：待处理信号队列
    
    def process_buy_signal(self, signal: Signal, execute_date: str):
        """处理买入信号"""
        current_holdings = len(self.positions)
        effective_max_positions = self.risk.get_effective_max_positions(execute_date)
        
        # 新增：over-cap 时加入待处理队列
        if current_holdings >= effective_max_positions:
            self.pending_signals.append({
                'signal': signal,
                'submit_date': execute_date,
                'priority': signal.strength
            })
            logger.info(f"Over-cap, added {signal.code} to pending queue")
            return self._reject_order(signal, "MAX_POSITIONS_REACHED_PENDING")
        
        # 正常买入
        return self._create_buy_order(signal, execute_date)
    
    def process_pending_signals(self, current_date: str):
        """每日检查待处理队列"""
        if not self.pending_signals:
            return
        
        current_holdings = len(self.positions)
        effective_max_positions = self.risk.get_effective_max_positions(current_date)
        
        # 如果有空位，处理待处理信号
        available_slots = effective_max_positions - current_holdings
        
        if available_slots > 0:
            # 按优先级排序
            self.pending_signals.sort(key=lambda x: x['priority'], reverse=True)
            
            # 处理前 N 个信号
            for i in range(min(available_slots, len(self.pending_signals))):
                pending = self.pending_signals.pop(0)
                
                # 检查信号是否仍然有效（不超过 3 天）
                days_elapsed = self._calculate_days_between(
                    pending['submit_date'], 
                    current_date
                )
                
                if days_elapsed <= 3:
                    logger.info(f"Processing pending signal: {pending['signal'].code}")
                    self._create_buy_order(pending['signal'], current_date)
                else:
                    logger.info(f"Pending signal expired: {pending['signal'].code}")
```

**预期效果**：
- 不强制平仓，保留现有持仓
- 等待自然退出后再接受新信号
- 风险：可能错过最佳入场时机

**方案 C：混合模式（推荐）**

```python
# 配置
MSS_OVERC AP_STRATEGY = "hybrid"  # "force_close" | "pending_queue" | "hybrid"

def process_buy_signal(self, signal: Signal, execute_date: str):
    """混合策略：根据信号强度决定"""
    current_holdings = len(self.positions)
    effective_max_positions = self.risk.get_effective_max_positions(execute_date)
    
    if current_holdings >= effective_max_positions:
        # 如果新信号非常强（strength > 0.8），强制腾空间
        if signal.strength > 0.8:
            weakest = self._find_weakest_position()
            if weakest and weakest.unrealized_pnl < 0:
                # 只平掉亏损持仓
                self._close_position(weakest, execute_date, "MAKE_ROOM_FOR_STRONG")
                return self._create_buy_order(signal, execute_date)
        
        # 否则加入待处理队列
        self.pending_signals.append({...})
        return self._reject_order(signal, "MAX_POSITIONS_REACHED_PENDING")
    
    return self._create_buy_order(signal, execute_date)
```

**预期效果**：
- 强信号优先，弱信号排队
- 平衡激进和保守
- 风险：参数调优复杂

def rank_candidates(self, candidates: List[StockCandidate], date: str) -> List[StockCandidate]:
    """对候选股票排序"""
    # 读取 IRS 评分
    irs_scores = self.store.read_irs_daily(date)
    
    # 诊断：打印 IRS 评分
    logger.debug(f"IRS scores: {irs_scores}")
    
    # 为每个候选股票计算综合评分
    for candidate in candidates:
        industry = candidate.industry
        irs_score = irs_scores.get(industry, 50)  # 默认 50
        
        # 诊断：检查权重
        logger.debug(f"Candidate {candidate.code}: industry={industry}, irs_score={irs_score}")
        
        # 计算综合评分
        candidate.final_score = (
            candidate.preselect_score * 0.3 +  # 初选评分
            irs_score * 0.7  # IRS 评分
        )
    
    # 排序
    candidates.sort(key=lambda x: x.final_score, reverse=True)
    
    return candidates

## 5. 风险控制

### 5.1 回退条件

1. **Phase 2 失败**：IRS 排序仍然无效
2. **Phase 3 失败**：长窗口 EV < 0
3. **Phase 4 失败**：无法超越 legacy_bof_baseline
4. **时间超期**：超过 10 周仍未完成

### 5.2 分阶段决策

每个 Phase 都有独立的 Go/No-Go 决策：
- Phase 1 失败 → 尝试其他 over-cap 方案
- Phase 2 失败 → 考虑移除 IRS 模块
- Phase 3 失败 → 重新评估架构假设
- Phase 4 失败 → 正式回退到 v0.01
