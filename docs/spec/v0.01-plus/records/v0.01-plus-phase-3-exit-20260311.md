## 3. 本轮回答的四个问题

### 3.1 容量变化来自真实状态变化，还是 fallback 路径

结论：`来自真实状态变化，不是 fallback 伪装。`

1. 本轮 `16 / 16` 条 overlay trace 的 `regime_source` 都是 `STORED`
2. `15 / 16` 条样本的 `overlay_reason = NORMAL`
3. 剩余 `1 / 16` 条样本是 `TREND_COLD_START`
4. 本轮正式证据里没有出现 `OVERLAY_MISSING` 或 `SNAPSHOT_MISSING`

### 3.2 `MarketScore.signal` 和 `risk_regime` 为什么不是一回事

结论：`因为 signal 只表达分数三态，risk_regime 表达 Broker 最终容量语义。`

代表性样本：

1. `2026-01-12`
   - `score = 66.0654`
   - `signal = BULLISH`
   - `phase = DIVERGENCE`
   - `phase_trend = UP`
   - `risk_regime = RISK_NEUTRAL`
2. `2026-01-09`
   - `score = 59.7313`
   - `signal = NEUTRAL`
   - `phase = ACCELERATION`
   - `phase_trend = UP`
   - `risk_regime = RISK_ON`
3. `2026-01-22` 与 `2026-02-04` 也重复出现 `signal = NEUTRAL` 但 `risk_regime = RISK_ON`

这说明 `Phase 3` 后 Broker 的正式消费面已经不是 `signal`，而是 `phase -> risk_regime`。

### 3.3 哪些天高分但必须降到 `RISK_OFF`

结论：`本轮短窗内没有出现这种样本。`

1. 在 `2026-01-05` 至 `2026-02-24` 内，没有出现 `score >= 60 且 risk_regime = RISK_OFF`
2. 这意味着本轮窗口里尚未打到 `CLIMAX -> RISK_OFF` 的代表性高位样本
3. 当前可确认的是：
   - 高分不必然 `RISK_ON`
   - `2026-01-12` 已经出现 `BULLISH -> RISK_NEUTRAL`

### 3.4 `signal` 与 `risk_regime` 判断相反时，Broker 的实际行为是什么

结论：`Broker 会按 risk_regime 的容量语义执行，而不是按 signal 三态执行。`

1. `RISK_OFF` 样本把有效仓位上限压到 `4`，并在：
   - `2026-01-29`
   - `2026-01-30`
   - `2026-02-05`
   - `2026-02-06`
   - `2026-02-13`
   上累计打出 `7` 条 `BROKER_CAPACITY_REJECT`
2. `RISK_ON` 样本恢复到基线容量：
   - `effective_max_positions = 10`
   - `effective_risk_per_trade_pct = 0.008`
   - `effective_max_position_pct = 0.10`
3. `2026-02-04` 的 `3` 条 `RISK_ON` 信号全部被接受，说明正式执行面已经切到 regime，而不是继续看 `signal`
