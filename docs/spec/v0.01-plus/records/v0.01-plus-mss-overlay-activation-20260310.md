## 1. 验证方式

本轮不新增长窗口脚本，直接复用已经存在的 `broker` 定向测试，验证三类最关键语义：

1. `overlay_state` 是否能真实进入 `DISABLED / MISSING / NORMAL`
2. `BEARISH` 下单笔头寸是否真的缩小
3. `BEARISH` 下有效仓位上限是否真的收缩

执行命令：

```powershell
pytest tests/unit/broker/test_broker.py -k "mss_overlay_reduces_effective_max_positions_under_bearish_regime or mss_overlay_reduces_position_size_under_bearish_regime or writes_mss_overlay_trace_for_disabled_missing_and_normal" -q
```

结果：

- `3 passed`

## 2. 已验证结论

### 2.1 overlay 确实能进入 `NORMAL`

`test_broker_writes_mss_overlay_trace_for_disabled_missing_and_normal` 已证明：

1. `DTT_VARIANT != MSS_RISK_OVERLAY_VARIANT` 时，写出 `DISABLED / OVERLAY_DISABLED`
2. 变体已启用但当日缺 `l3_mss_daily` 快照时，写出 `MISSING / SNAPSHOT_MISSING`
3. 变体已启用且当日存在 `l3_mss_daily` 快照时，写出 `NORMAL`
4. `NORMAL` 样本能保留：
   - `market_signal = BEARISH`
   - `ranker_mss_score = 20.0`

结论：`MSS overlay` 不再只是配置名或 trace 名义存在，当前主线已能真实进入执行态 `NORMAL`。

### 2.2 `BEARISH` 下单笔容量会缩小

`test_broker_mss_overlay_reduces_position_size_under_bearish_regime` 已证明：

1. `BULLISH` 时下单数量 = `50000`
2. `BEARISH` 时下单数量 = `20000`
3. 比例 = `0.4`

这个结果与当前 `MSS_BEARISH_RISK_PER_TRADE_MULT=0.4`、`MSS_BEARISH_MAX_POSITION_MULT=0.4` 的配置口径一致。

### 2.3 `BEARISH` 下有效仓位上限会收缩

`test_broker_mss_overlay_reduces_effective_max_positions_under_bearish_regime` 已证明：

1. 基线 `MAX_POSITIONS=3`
2. `BULLISH` 下仍可接受新单
3. `BEARISH` 下有效上限收缩到 `1`
4. 在已有 `2` 个持仓时，同一信号被拒绝，理由为 `MAX_POSITIONS_REACHED`

结论：`MSS overlay` 当前已经真实影响执行层的“能不能开仓”，不只是影响 trace。
