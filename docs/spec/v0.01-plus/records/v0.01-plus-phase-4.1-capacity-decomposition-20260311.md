## 4. 核心归因结论

### 4.1 主嫌疑已经收口到 `max_positions`

`only_max_positions` 单独就复现了 `full_overlay` 的关键坏结果：

1. `trade_count`：`9 == 9`
2. `expected_value`：`-0.008988 == -0.008988`
3. `profit_factor`：`0.4613 == 0.4613`
4. `reject_rate`：`0.2800 == 0.2800`
5. `participation_rate`：`0.5625 == 0.5625`
6. `buy_reject_maxpos_count`：`7 == 7`

对照层面也一致：

1. `baseline_no_overlay vs only_max_positions`
   - `trade_set_changed_count = 7`
   - `quantity_changed_count = 2`
   - `reject_set_changed_count = 7`
2. `full_overlay vs only_max_positions`
   - `trade_set_changed_count = 0`
   - `reject_set_changed_count = 0`
   - 只剩 `quantity_changed_count = 7`

约束性结论：

`当前正式短窗 NO-GO 路径的第一主嫌疑，不是 sizing 杠杆，而是 slot 缩容杠杆 max_positions。`

### 4.2 `risk_per_trade` 和 `max_position_pct` 是二级影响，不是主嫌疑

两者的共同特征：

1. 相对 `baseline_no_overlay`：
   - `trade_set_changed_count = 0`
   - `reject_set_changed_count = 0`
2. 它们只改了仓位数量，不改成交集合

具体表现：

1. `only_risk_per_trade`
   - `quantity_changed_count = 13`
   - `MDD: 0.017079 -> 0.012121`
2. `only_max_position_pct`
   - `quantity_changed_count = 14`
   - `MDD: 0.017079 -> 0.009241`

约束性结论：

`risk_per_trade / max_position_pct` 在当前窗口主要贡献的是“缩量降波动”，而不是“删交易集合”。`

### 4.3 full overlay 的坏结果来自“两层叠加”，但主导层级已经清楚

`full_overlay` 相对 `only_max_positions`：

1. `trade_set_changed_count = 0`
2. `reject_set_changed_count = 0`
3. `quantity_changed_count = 7`
4. `MDD: 0.009400 -> 0.006035`

约束性结论：

1. `max_positions` 决定了“哪些票被挡掉、交易数为什么从 16 变成 9”
2. `risk_per_trade / max_position_pct` 决定了“留下来的 9 笔单子被缩成多大”
3. 因此当前整改顺序必须先拆 slot path，再拆 sizing path
