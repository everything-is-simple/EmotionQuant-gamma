# v0.01-plus 数据契约表（主线迁移补充）

- **版本**: `v0.01-plus 契约补充`
- **状态**: `Active`
- **日期**: `2026-03-07`
- **基线**: `docs/design-v2/01-system/system-baseline.md`, `docs/spec/v0.01/governance/v0.01-data-contract-table.md`, `docs/spec/v0.01-plus/governance/v0.01-plus-run-artifact-rules.md`

## 1. 原则

1. `v0.01 Frozen` 的正式跨模块契约继续保留为历史基线与回退参考。
2. `v0.01-plus` 是当前主开发线；当前阶段采用“兼容优先”的迁移策略，先不强制让下游与 formal schema 同步大改。
3. 缺失策略必须显式声明为 `FAIL / SKIP / FILL / REJECT` 之一。

## 2. 核心契约（v0.01-plus）

| 表名 | 用途 | 主键/幂等键 | As-Of 规则 | 缺失策略 | 口径备注 |
|---|---|---|---|---|---|
| `l3_mss_daily` | 市场情绪分数 | 沿用 `v0.01` | 沿用 `v0.01` | 沿用 `v0.01` | 正式表，不因主线替代改表 |

## 3. `l3_signal_rank_exp` 字段建议

| 字段 | 类型 | 说明 |
|---|---|---|
| `run_id` | `VARCHAR` | 对应 `_meta_runs.run_id` |
| `signal_id` | `VARCHAR` | 对应正式 `l3_signals.signal_id` |
| `signal_date` | `DATE` | 对应正式信号日 |
| `code` | `VARCHAR` | 对应正式 `code` |
| `industry` | `VARCHAR` | 候选所属行业，缺失记 `未知` |
| `variant` | `VARCHAR` | 四组切换矩阵之一 |
| `bof_strength` | `DOUBLE` | BOF 强度原始值或标准化值 |
| `irs_score` | `DOUBLE` | 行业后置分；缺失时 `50.0` |
| `mss_score` | `DOUBLE` | 市场后置分；缺失时 `50.0` |
| `final_score` | `DOUBLE` | 当前主线规则下的最终排序分 |
| `final_rank` | `INTEGER` | 同一 run 的最终排序 |
| `selected` | `BOOLEAN` | 是否进入当前主线的最终执行集合 |
| `created_at` | `TIMESTAMP` | 写入时间 |

## 4. 缺失策略

1. `signal_id` 缺失：`FAIL`
2. `run_id` 缺失：`FAIL`
3. `bof_strength` 缺失：`FAIL`
4. `irs_score` 缺失：`FILL=50.0`
5. `mss_score` 缺失：`FILL=50.0`
6. 正式表 schema 不一致：`FAIL`

## 5. 兼容性约束

1. 第一阶段 `Broker / Matcher / Report` 继续只消费兼容期正式 `Signal / Order / Trade` 契约。
2. 当前主线的排序明细统一通过 `run_id + signal_id` 追溯，不要求下游模块立即理解 `DTT` 评分字段。
3. 若后续决定把 `bof_strength / irs_score / mss_score / final_score` 正式并入 `Signal`，必须单独补 schema migration 与 compatibility 方案。
