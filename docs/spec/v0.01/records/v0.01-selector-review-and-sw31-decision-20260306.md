# v0.01 Selector 评审与 SW31 口径决策

日期：`2026-03-06`

## 1. 结论

本轮评审确认三件事：

1. 当前执行代码此前使用的并不是申万一级 `31` 行业，而是 `stock_basic.industry` 快照字段，对应现实桶数约 `110/111`。
2. MSS 当前的核心问题不是“没有标准化”，而是 `baseline` 仍为占位参数；`zscore -> 0~100` 已实现，但 `mean/std` 未校准。
3. IRS 的异常低覆盖并不全是排序算法问题，执行库里还混入了非交易日/样例污染数据，必须从数据层先止血。

因此，v0.01 当前决策为：

`行业分类口径收敛到 SW2021 一级 31 行业；申万成员映射优先，stock_basic.industry 仅保留兼容兜底。`

---

## 2. 官方口径确认

本轮以 TuShare 官方文档为准：

- `index_classify`
  定义申万行业分类；`SW2021` 版本为 `31` 个一级行业。
  参考：[TuShare 文档 doc_id=181](https://tushare.pro/document/2?doc_id=181)
- `index_member_all`
  提供申万行业成分构成及 `l1_code / l1_name / ts_code / in_date / out_date`。
  参考：[TuShare 文档 doc_id=335](https://tushare.pro/document/2?doc_id=335)

这意味着：

- `stock_basic.industry` 不是申万一级行业事实源
- 如果系统设计写的是“申万一级 31 行业”，运行时就必须通过 `index_classify + index_member` 链路实现

---

## 3. 当前发现

### 3.1 旧实现的行业口径是错的

旧实现链路为：

`stock_basic.industry -> l1_stock_info.industry -> clean_industry_daily() -> l2_industry_daily`

实际执行库抽样结果显示行业桶名称为：

- `元器件`
- `专用机械`
- `汽车配件`
- `软件服务`

这与申万一级 `31` 行业明显不一致。

### 3.2 MSS 的问题是 baseline，不是缺 zscore

当前代码中：

- `src/selector/normalize.py` 已实现 `zscore -> 0~100`
- `src/selector/baseline.py` 的 `MSS_BASELINE` 仍为 `mean=0/std=1` 占位参数

所以 MSS 现在“分布压缩、长期偏中性”的主要原因是：

`baseline 未校准`

不是“完全没做归一化”。

### 3.3 执行库存在非交易日/样例污染

执行库中出现了以下异常：

- `2026-02-14 ~ 2026-02-17` 在交易日历中均为 `is_trade_day = false`
- 这些日期却存在 `l1_stock_daily` 行
- 个别行对应乱码名称/乱码行业

因此此前 `industries_per_day.min = 1` 的异常，不能直接全部归因于 IRS 算法；其中一部分是在给脏数据报警。

### 3.4 raw 源库的申万成员数据也不完整

虽然 raw 源库中存在：

- `raw_index_classify`
- `raw_index_member`

但真实校验显示：

- `raw_index_classify` 内同时混有真实申万数据和样例污染数据
- 过滤样例后，真实 `SW2021 L1` 仍可恢复出 `31` 个行业
- 但 `raw_index_member` 最新快照中的真实一级行业覆盖明显不完整

本轮短窗验证库结果：

- `l2_industry_daily` 仅收敛到 `27 + 未知`
- `l3_irs_daily` 每日固定 `27`

因此：

`申万口径已经修正，但 raw 申万成员源数据仍需补采/刷新，当前还不能宣布 IRS 数据源完全收口。`

---

## 4. 已落地改动

本轮已落地以下修正：

1. 新增 `l1_sw_industry_member`
   作为申万一级成员映射表。
2. raw bootstrap 新增申万成员导入
   从 `raw_index_classify + raw_index_member` 导入到执行库。
3. `clean_stock_adj_daily()` 只处理交易日
   非交易日污染不再进入 L2。
4. `clean_industry_daily()` 优先使用申万成员映射
   仅在缺映射时回退到 `stock_basic.industry`。
5. `selector.py` 的个股行业归属改为申万优先
   保证 `IRS` 排名与候选池行业字段口径一致。
6. `compute_irs()` 增加最小行业数门槛
   运行时通过 `IRS_MIN_INDUSTRIES_PER_DAY` 控制，当前默认 `25`。

## 5. 决策

### 5.1 v0.01 运行时行业口径

正式定为：

`SW2021 一级 31 行业`

### 5.2 行业映射优先级

正式定为：

1. `l1_sw_industry_member`
2. `l1_stock_info.industry`（仅兼容兜底）

因此，`stock_basic.industry` 不再是事实源，只是过渡兼容字段。

### 5.3 IRS 日级有效性契约

正式定为：

- 非交易日不参与 `L2/L3`
- `industries_per_day < IRS_MIN_INDUSTRIES_PER_DAY` 时，当日 IRS 视为无效，不写入 `l3_irs_daily`
- 行业排名必须唯一、稳定、可复现

### 5.4 design-v2 的处理建议

当前建议不是“全面解冻”，而是：

`打开一条受控纠偏轨道，只修正算法权威入口与数据分类口径，不改 v0.01 执行语义。`

换句话说：

- `BOF only`
- `T+1 Open`
- 模块边界
- 结果契约

这些继续冻结；

但 `MSS / IRS / PAS` 的算法权威入口，以及 `IRS` 的申万口径，需要从“散落在 records/reference”收回到正式设计层。

## 6. 下一步

### 6.1 Data

优先刷新 raw 源库中的申万行业数据：

- 清洗 `raw_index_classify` 的样例污染
- 用官方 TuShare 通道补齐/刷新 `raw_index_member`

### 6.2 MSS

优先做 `baseline calibration`：

- 从真实 `l2_market_snapshot` 统计六因子 `mean/std/p05/p25/p50/p75/p95`
- 替换占位式 `MSS_BASELINE`
- 重跑阈值敏感性审计（`55/58/60/62/65`）

### 6.3 设计治理

补一组正式算法设计入口，避免后续 SoT 继续分裂：

- `design-v2/mss-algorithm.md`
- `design-v2/irs-algorithm.md`
- `design-v2/pas-algorithm.md`

如果不愿在 `design-v2` 直接扩写，则至少要明确：

`system-baseline = 系统 SoT`

`算法文档 = 设计级 SoT`

`records = 证据与阶段决策`

不能继续让三者边界混杂。
