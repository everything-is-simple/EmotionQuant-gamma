# v0.01 当前版算法设计：PAS

日期：`2026-03-06`

## 1. 结论先说

PAS 是当前系统里最健康的一块算法设计。

原因不是它最复杂，而是它最清楚：

`PAS = detector registry + pattern detectors`

这条路比旧版 `core-algorithms` 那种“一个 PAS 总分系统”更适合当前系统。

## 2. 为什么当前 PAS 设计更好

旧版 `core-algorithms` 的 PAS 更像：

- 个股机会打分
- 机会等级分层
- 风险收益比评估

这个方向容易做成一个“大而全评分器”，但对当前系统不够直接。

当前系统真正需要的是：

1. 候选池已经有了
2. 现在只回答“今天这只票有没有形态触发”
3. 触发后由 Broker 处理仓位和退出

所以 PAS 最自然的形态就是：

`多个 detector 统一注册，按配置装配。`

## 3. 当前实现文件

- `src/strategy/pattern_base.py`
- `src/strategy/pas_bof.py`
- `src/strategy/registry.py`
- `src/strategy/strategy.py`

## 4. 当前架构定义

### 4.1 PatternDetector 抽象基类

当前统一约束是：

- 每个形态一个 detector
- 输入统一
- 输出统一

也就是：

```text
detect(code, asof_date, df) -> Signal | None
```

这让每个形态都可以：

- 单独回测
- 单独开关
- 单独写单测

### 4.2 registry 负责装配，不负责计算

`registry.py` 的职责不是算信号，而是：

- 管理 detector 名单
- 按配置决定当前启用哪些 detector

这点非常重要，因为它让：

`算法定义`

和

`运行时装配`

彻底分开了。

### 4.3 strategy.py 负责汇总，不负责形态细节

`strategy.py` 做的是：

1. 为每只候选票准备历史窗口
2. 调用活跃 detector
3. 合并多 detector 结果
4. 幂等写入 `l3_signals`

它不应该掺进具体 BOF/BPB/PB 的判定细节。

## 5. v0.01 当前口径

### 5.1 只启用 BOF

当前 v0.01 的强约束是：

- 只允许 `bof`
- 不允许多形态并行启用

这个限制现在已经在实现上做了强校验。

### 5.2 BOF 当前定义

当前 BOF 触发条件是：

1. `low < lower_bound * (1 - break_pct)`
2. `close >= lower_bound`
3. `close_pos >= 0.6`
4. `volume >= volume_ma20 * volume_mult`

满足即在 `T` 日收盘后生成 BUY 信号。

### 5.3 SELL 不属于 PAS

这也是当前设计非常对的一点：

- PAS 只产 BUY
- SELL 由 Broker 风控层生成

这样 detector 保持纯粹，避免把触发器和持仓管理糊在一起。

## 6. 当前 PAS 的开放点

### 6.1 架构已经对，扩展还没到位

当前 `registry.py` 还没有把 `bpb / tst / pb / cpb` 作为占位 detector 真正注册进去，只保留了注释位。

这不影响 v0.01，但说明：

`架构已经成型，生态还没补齐。`

### 6.2 组合模式已经在位，但 v0.01 不该急着用

当前 `strategy.py` 已支持：

- `ANY`
- `ALL`
- `VOTE`

但 v0.01 只应该用：

- `bof`
- `ANY`

因为在只有一个 detector 时，组合模式不是主问题。

## 7. 我建议的文档定性

对于 PAS，我建议把当前设计正式定性为：

`PAS 不是单一评分算法，而是一个形态触发器框架。`

在这个框架里：

- `BOF` 是 v0.01 唯一在线 detector
- `BPB` 是 v0.02 第一优先扩展
- `TST / PB / CPB` 后续逐步接入

## 8. v0.02+ 的正确扩展方式

后面如果要加新形态，我建议继续坚持当前套路：

1. 一形态一 detector
2. 先单形态独立回测
3. 通过后再注册进组合
4. 不把 MSS/IRS 分数塞进 detector 输入

这条路是当前系统里最值得保留的设计资产。

## 9. 本文档的明确结论

MSS 和 IRS 当前都还在“校准期”。

但 PAS 不一样。

PAS 当前的主问题不是架构，而是版本范围：

- v0.01 只跑通 `BOF`
- 后续形态按 detector 模式继续接入

所以我对 PAS 的建议不是“重做”，而是：

`保持 detector registry 架构不动，把它当作当前系统的算法主骨架。`
