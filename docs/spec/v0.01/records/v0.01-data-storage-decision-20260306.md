# v0.01 数据存储口径决议（2026-03-06）

**日期**: 2026-03-06  
**适用版本**: v0.01  
**状态**: Active  
**目的**: 结束“执行库 / raw 源库 / Parquet / logs / cache”并存但口径不清的状态，统一当前实现与运维实践。

---

## 1. 结论

v0.01 起，数据存储统一采用“**一个执行库 + 一个 raw 源库 + 非主链路辅助目录**”模型，不再保留“DuckDB + Parquet 双主存储”口径。

### 1.1 唯一执行库

`G:\EmotionQuant_data\emotionquant.duckdb`

定位：

1. 当前系统唯一执行库。
2. `main.py fetch/build/run/backtest` 的默认读写目标。
3. L1/L2/L3/L4 与 `_meta_*` 表统一落在此库。

约束：

1. 运行时只认这一库，不允许再引入第二个“执行库”概念。
2. 任何回测、联调、报告、纸上交易的结果，默认都写入这一库。

### 1.2 raw 历史源库

`G:\EmotionQuant_data\duckdb\emotionquant.duckdb`

定位：

1. 本地 raw 历史数据源库。
2. 用于 bootstrap、历史回填、证据构建、change-point 补齐。
3. 不作为运行时主库，不直接承担 Selector / Strategy / Broker / Report 的在线读写。

约束：

1. 只能作为源库被映射/导入到执行库。
2. 当前主入口已支持：`python main.py fetch --from-raw-db <path>`。

### 1.3 Parquet

`G:\EmotionQuant_data\parquet`

定位：

1. 旧版遗留冷备与归档格式。
2. 当前仅保留 `parquet/l1` 作为历史沉淀，不再作为 v0.01 主链路 SoT。

约束：

1. v0.01 不再维持“DuckDB + Parquet 双写、双读、双主存储”。
2. Parquet 不参与 `main.py` 主流程的默认读写。
3. 若后续继续保留，只能作为备份、迁移中转或离线交换格式。

---

## 2. 目录职责

### 2.1 logs

`G:\EmotionQuant_data\logs`

定位：

1. 运行日志目录。
2. 存放 CLI 运行日志、错误日志、必要的 run 级排障日志。

规则：

1. 不写业务数据。
2. 不写缓存。
3. 不作为状态源。

### 2.2 cache

`G:\EmotionQuant_data\cache`

定位：

1. 临时缓存目录。
2. 用于下载中间物、探针文件、临时导出、可重建的一次性产物。

规则：

1. `cache` 不是系统事实源。
2. 其中内容应允许随时删除后重建。
3. 不允许把关键业务状态只保存在 `cache`。

---

## 3. 统一口径

当前 v0.01 的正式口径为：

1. **执行**：单执行库 `G:\EmotionQuant_data\emotionquant.duckdb`
2. **历史 raw 源**：`G:\EmotionQuant_data\duckdb\emotionquant.duckdb`
3. **Parquet**：冷备/遗留，不参与主链路
4. **logs**：日志
5. **cache**：临时缓存

换句话说，当前是“**两个 DuckDB，但不是双主库**”，而是：

`执行库 + raw 源库`

不是：

`两个执行库`

也不是：

`DuckDB + Parquet 双主存储`

---

## 4. 对实现的要求

1. `src/config.py` 中 `db_path` 代表唯一执行库。
2. `RAW_DB_PATH` 只用于 bootstrap / raw 导入。
3. `duckdb_dir` 与 `parquet_path` 仅保留兼容旧脚本的辅助路径语义，不得再被解释为在线主存储。
4. 后续文档、脚本、runbook 一律按本决议表述，不再混用“主库”“缓存库”“Parquet 主存储”等旧口径。

---

## 5. 后续动作

2. Week2 起：所有实现默认围绕唯一执行库推进，不再设计 Parquet 主链路。
3. 若未来需要删除 Parquet 遗留路径，应作为单独整理任务处理，不影响 v0.01 主语义。
