## 3. 当前缺口

### G1. 本地 `raw_*` 历史库优先仍是脚本能力，不是主链路能力

现状：
- 仓库已有 `scripts/data/load_l1_from_raw_duckdb.py`
- `main.py fetch` 仍直接走 API 抓取链路，没有“优先映射本地 raw 库”的内建入口

影响：
- 文档已经把“本地历史库优先”写成 v0.01 现实主链路，但 CLI 主入口尚未内化这条能力。

建议：
- 下一步把 raw 库导入包装成正式命令入口，至少做到：
  - `python main.py fetch --from-raw-db <path>`
  - 或新增 `python main.py bootstrap-l1-from-raw`

### G2. `stock_info` 的 change-point 历史仍依赖 raw 快照库

现状：
- `TuShareFetcher.fetch_stock_info()` 现在能保留 `list_status`
- 但 TuShare 当前接口仍只提供“当前快照”，不能单靠 API 回溯完整历史变更点

影响：
- 对“从零重建历史 as-of 状态”来说，API 路径依然不充分。

建议：
- 保持当前修正，同时明确：历史重建仍以 raw 快照导入为准；API 路径只负责增量补数与当前状态补齐。

### G3. `fetch_incremental` 只有表级断点，没有批次级断点

现状：
- `_meta_fetch_progress` 只记录 `data_type -> last_success`
- `stock_daily` 虽按月切片抓取，但中途失败时不会记录到“已完成到哪一个月”

影响：
- 大窗口抓取失败后，需要从整段窗口重试，而不是从最后成功切片继续。

建议：
- 若继续强化 v0.01，可把断点粒度下沉到 `(data_type, slice_end)` 或运行级 checkpoint。

### G4. `spec-01` 里写到的并行拉取尚未实现

现状：
- 代码未见 `ThreadPoolExecutor`
- `main.py fetch` 仍按 `trade_cal -> stock_info -> index_daily -> stock_daily` 顺序拉取

影响：
- 这不是口径错误，但与实现卡承诺不一致。

建议：
- 若优先级不足，可先把实现卡改成“可延后优化项”；否则应补到 fetch 主链路。

---

## 4. 审计结论

1. Week1 Data Layer 的基础骨架已经能跑，不是“未实现”状态。
2. 当前更准确的任务定义不是“从零实现 Data Layer”，而是“把实战主链路能力收口进正式 CLI”。
3. 下一轮应优先处理 `G1`，其次再决定是否在 v0.01 内处理 `G3/G4`。
