把沙盘模拟从“经验排查”升级为“可重复执行的评审流程”，避免单点修复后在调用链其他位置再次失效。

本标准是 `system-baseline.md` 的配套评审规范；若与主文档冲突，以 `system-baseline.md` 为准。

## 2. 七维评审框架（固定口径 + 可执行检查清单）

### 2.1 Schema 契合（骨骼）

必查清单：

1. DDL 字段与 pydantic 契约字段一一对应（字段名、类型、nullable、默认值）。
2. 关键语义字段强约束一致：`pattern`、`execute_date`、`status`、`is_paper`。
3. SQL 查询引用字段必须存在于目标表，禁止“文档有、DDL无”。
4. 索引与主键是否覆盖幂等键（`signal_id`/`order_id`/`trade_id`）。
5. 变更后必须有一次“schema diff 复核”记录。

通过判定：

1. Schema 对齐缺陷 `S0/S1 = 0`。
2. 关键表（L3/L4）字段对齐清单完整。

### 2.2 调用链完整性（神经）

必查清单：

1. 主链路逐层签名核对：`next -> process_signals -> check_signal -> matcher.execute`。
2. 深层函数依赖参数必须显式透传（如 `today_date`、`broker_state`）。
3. 禁止隐式使用 `date.today()` 替代回测时间。
4. 每条调用链至少一个“参数缺失即失败”的断言/单测。
5. 关键调用图（函数名+参数）需随修复同步更新。

通过判定：

1. 无 `TypeError/NameError/missing argument` 路径。
2. 回测与纸上交易同一入口参数语义一致。

### 2.3 幂等与确定性（免疫）

必查清单：

1. 关键 ID 必须确定性：`signal_id/order_id/trade_id`。
2. 可重跑表统一 `upsert`，禁止 `insert-only` 追加脏数据。
3. “同输入重跑两次”结果一致（行数、主键集合、关键指标）。
4. 强平与风控生成单也必须走确定性键规则。
5. 明确“覆盖而非追加”的实现注释与测试。

通过判定：

1. 重跑后主键去重率 100%。
2. 无幽灵订单、重复成交。

### 2.4 状态机完整性（内分泌）

必查清单：

1. 所有状态迁移可达：`ACTIVE -> OBSERVE -> BACKUP -> OBSERVE -> ACTIVE`。
2. 升降级触发条件与调用点一致（文档、代码、测试三对齐）。
3. `update_trust` 触发事件明确（v0.01 仅 `SELL` 触发）。
4. 试用期字段（`on_probation`）在 DDL/逻辑/文档一致。
5. 冷却/自动升级路径必须有时间推进测试。

通过判定：

1. 无死路径、无永不恢复路径。
2. 状态转移图与实现一致。

### 2.5 时序语义（心跳）

必查清单：

1. 固定语义：`signal_date=T -> execute_date=T+1 -> price=T+1 open`。
2. 撮合入口断言：`status=PENDING && execute_date==today`。
3. 禁止 T 日 close 成交（未来函数）。
4. 涨跌停/停牌可成交检查使用一致价格口径（原始价 vs 限价）。
5. 回测末日强平例外必须在文档显式声明并与统计口径一致。

通过判定：

1. 时序相关 `S0=0`。
2. 回测与纸上交易时序规则一致，例外路径仅限终止结算。

### 2.6 边界冲突与去重（排异）

必查清单：

1. 同日同标的多规则并发触发时只允许一笔有效 SELL。
2. 组合回撤与个股止损同时触发时有优先级和去重机制。
3. 禁止负仓位、超卖、重复撮合。
4. 风险熔断与清仓并发时有明确顺序。
5. 冲突场景必须有回归用例（急跌日、停牌日、一字板）。

通过判定：

1. `_pair_trades` 无库存负数错误。
2. 边界冲突下订单/成交数量一致。

### 2.7 报告口径真实性（体检）

必查清单：

1. `pnl/pnl_pct` 必须为净口径（手续费+滑点后）。
2. BUY/SELL FIFO 数量配对支持分批成交/分批平仓。
3. 报告不得二次错误配对（已配对数据不可重复配对）。
4. 强平成交需纳入统一成本口径。
5. 报告结论必须含样本量与环境分桶说明。

通过判定：

1. 指标可回溯到交易明细，且链路闭环。
2. 报告不因配对或成本口径错误失真。

## 3. 连环失效检测（跨维度必做）

原则：单点看似“低危”的偏差，可能跨维度联动导致子系统整体失效。

必做步骤：

1. 构建“偏差依赖图”（节点=偏差项，边=影响关系）。
2. 识别 kill-chain：任意 2+ 偏差组合导致核心能力失效，按 `S0` 处理。
3. 每轮修复后回放前一轮 kill-chain，确认不复发。
4. 对关键子系统（Broker 信任、撮合、报告）必须有组合场景测试。

本项目已验证的典型案例：

1. `Q41 + Q42 + R43 + S44` 联合失效，导致信任分级子系统形同死代码。
2. 该类问题在本标准下按 `S0` 处理，不可拆分为多个 `S2` 延后。

## 4. 证据模板（每个偏差必须填写）

每个偏差必须包含以下 8 项：

1. 偏差编号（如 `Q41`）
2. 触发条件（什么输入会触发）
3. 失败表现（报错/错误行为）
4. 影响范围（模块、数据表、状态机路径）
5. 根因定位（具体函数与文件行）
6. 修复点（改动的函数、字段或断言）
7. 回归断言（如何证明修复有效且不回归）
8. 关联偏差（是否属于连环失效链）

## 5. 风险分级（固定）

1. `S0-阻断`：会导致崩溃、未来函数、数据污染、状态机失效、连环失效
2. `S1-高危`：结果显著偏差，可能误导交易决策
3. `S2-中危`：局部功能失效或可观测偏差
4. `S3-低危`：文档/注释/可维护性问题

处置要求：

1. `S0/S1` 必须在定稿前清零。
2. `S2` 必须有明确 owner 与截止日期。
3. `S3` 可进入后续迭代，但需登记。

## 6. 定稿门禁（Go/No-Go）

满足以下条件才可判定“可定稿”：

1. 七维检查全部完成，且有证据记录。
2. `S0/S1 = 0`。
3. 连环失效检测完成，kill-chain 回放通过。
4. 三组消融链路完整可复现：`BOF baseline -> BOF+MSS -> BOF+MSS+IRS`。
5. 幂等重跑通过（同输入重跑不新增脏数据）。
6. T+1 语义与末日强平例外在文档中显式并列。

当前主开发线补充：

- 当前主开发线允许把消融链路替换为 `legacy_bof_baseline / v0_01_dtt_bof_only / v0_01_dtt_bof_plus_irs_score / v0_01_dtt_bof_plus_irs_mss_score`
- 但评审方法、证据要求和风险分级规则保持不变
