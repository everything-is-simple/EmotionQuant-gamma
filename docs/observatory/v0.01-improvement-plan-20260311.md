# v0.01 改进方案（保守路线）

**文档版本**: v1.0  
**创建日期**: 2026-03-11  
**状态**: Draft  
**适用场景**: 希望在已验证的 v0.01 架构基础上做渐进式改进

---

## 1. 方案定位

### 1.1 核心理念

**保持 v0.01 的前置过滤架构，但优化过滤强度，避免过度保守。**

### 1.2 设计原则

1. **架构不变**：MSS/IRS 继续作为前置漏斗
2. **参数优化**：调整阈值，增加机会捕获率
3. **渐进验证**：每次只改一个参数，对比验证
4. **风险可控**：保留回退路径

---

## 2. v0.01 的核心问题诊断

### 2.1 问题 1：MSS 硬门控过于保守

**现状**：
```
MSS 65 阈值下（2023-01-01 到 2026-02-24）：
- BULLISH = 20 天（2.6%）
- BEARISH = 23 天（3.0%）
- NEUTRAL = 714 天（94.3%）
```

**问题**：
- 94% 的时间系统处于"观望"状态
- 大量潜在机会被硬门控拒之门外
- 即使市场有结构性机会，系统也不参与

**根因**：
- MSS 作为"硬门控"时，必须非常保守
- 65 阈值设置过高，导致几乎不触发

---

### 2.2 问题 2：IRS 硬过滤容易误杀

**现状**：
```
IRS 作为前置硬过滤：
- 必须设定 Top N 行业
- 非 Top N 行业的股票直接被删除
- 无法适应不同市场环境
```

**问题**：
- 如果 N=5，可能错过其他 26 个行业的机会
- 如果 N=15，过滤就失去意义
- "一刀切"的过滤方式缺乏灵活性

**根因**：
- IRS 作为"硬过滤"时，必须做强制排他
- 无法根据市场状态动态调整

---

### 2.3 问题 3：样本量不足导致过拟合风险

**现状**：
```
v0.01 (2023-01-01 到 2026-02-24):
- 交易数：256 笔
- 平均每月：7 笔
- 平均每周：1.6 笔
```

**问题**：
- 样本量太小，统计显著性不足
- 容易受到个别极端案例影响
- 难以验证策略的稳定性

**根因**：
- MSS/IRS 硬门控删除了大量机会
- 导致最终成交样本过少

---

## 3. 改进方案

### 3.1 改进 1：MSS 阈值优化（优先级 P0）

#### 3.1.1 目标

将 MSS BULLISH 触发率从 2.6% 提升到 15-25%，增加机会捕获。

#### 3.1.2 具体方案

**方案 A：降低 BULLISH 阈值**

```python
# 当前配置
MSS_BULLISH_THRESHOLD = 65
MSS_BEARISH_THRESHOLD = 35

# 改进配置
MSS_BULLISH_THRESHOLD = 55  # 从 65 降到 55
MSS_BEARISH_THRESHOLD = 35  # 保持不变
```

**预期效果**（基于历史校准数据）：
- BULLISH 天数：20 → 165（21.8%）
- 机会增加：8 倍
- 风险：可能引入更多假信号

**验证方法**：
1. 回测 2023-01-01 到 2026-02-24
2. 对比 threshold=55/58/60/62/65 的表现
3. 选择 EV/PF/MDD 综合最优的阈值

---

**方案 B：引入"软门控"模式**

```python
# 新增配置
MSS_GATE_MODE = "soft"  # "hard" | "soft"

# 软门控逻辑
if MSS_GATE_MODE == "soft":
    if mss_signal == "BEARISH":
        # 只在 BEARISH 时硬拒绝
        reject_all_signals()
    else:
        # NEUTRAL 和 BULLISH 都允许通过
        pass
else:
    # 硬门控（当前模式）
    if mss_signal != "BULLISH":
        reject_all_signals()
```

**预期效果**：
- BULLISH + NEUTRAL 天数：734 天（97%）
- 只在明确 BEARISH 时拒绝
- 风险：NEUTRAL 时可能表现不佳

**验证方法**：
1. 对比 hard gate vs soft gate
2. 分析 NEUTRAL 天数的交易表现
3. 如果 NEUTRAL 天数 EV < 0，则放弃此方案

### 3.2 改进 2：IRS 动态 Top N（优先级 P1）

#### 3.2.1 目标

将 IRS 从"固定 Top N"改为"动态 Top N"，根据市场状态调整。

#### 3.2.2 具体方案

**方案 A：基于 MSS 的动态 Top N**

```python
# 当前配置
IRS_TOP_N = 10  # 固定值

# 改进配置
def get_dynamic_top_n(mss_signal: str) -> int:
    """根据 MSS 信号动态调整 IRS Top N"""
    if mss_signal == "BULLISH":
        return 15  # 牛市时放宽，捕获更多行业
    elif mss_signal == "BEARISH":
        return 5   # 熊市时收紧，只选最强行业
    else:
        return 10  # 中性时保持默认
```

**预期效果**：
- 牛市时不错过次强行业
- 熊市时只选最强行业
- 适应性更强

---

**方案 B：基于行业强度分布的动态 Top N**

```python
def get_adaptive_top_n(industry_scores: List[float]) -> int:
    """根据行业强度分布动态调整 Top N"""
    # 计算 Top 5 和 Top 10 的平均分差
    top5_avg = mean(industry_scores[:5])
    top10_avg = mean(industry_scores[5:10])
    
    score_gap = top5_avg - top10_avg
    
    if score_gap > 10:
        # 强度分化明显，只选 Top 5
        return 5
    elif score_gap < 5:
        # 强度分化不明显，扩大到 Top 15
        return 15
    else:
        # 中等分化，保持 Top 10
        return 10
```

**预期效果**：
- 自动适应行业轮动强度
- 避免"一刀切"

### 3.3 改进 3：增加"候选池扩容"机制（优先级 P2）

#### 3.3.1 目标

在不改变 MSS/IRS 硬门控的前提下，增加候选池规模，提高样本量。

#### 3.3.2 具体方案

**方案 A：放宽基础过滤条件**

```python
# 当前配置
MIN_AMOUNT = 50000  # 千元，即 5000 万
MIN_VOLUME = 1000000  # 100 万股

# 改进配置
MIN_AMOUNT = 30000  # 降低到 3000 万
MIN_VOLUME = 500000  # 降低到 50 万股
```

**预期效果**：
- 候选池从 200 扩大到 300-400
- 增加中小市值股票
- 风险：流动性可能不足

---

**方案 B：增加"备选池"机制**

```python
# 新增配置
ENABLE_BACKUP_POOL = True
BACKUP_POOL_SIZE = 50

# 逻辑
def select_candidates(all_stocks):
    # 主池：通过 MSS/IRS 硬门控
    main_pool = apply_mss_irs_filter(all_stocks)
    
    if ENABLE_BACKUP_POOL:
        # 备选池：未通过硬门控，但基础指标良好
        backup_pool = get_backup_candidates(all_stocks, size=50)
        
        # 合并，但标记来源
        return main_pool + backup_pool
    else:
        return main_pool
```

**预期效果**：
- 增加样本多样性
- 不完全依赖 MSS/IRS
- 可以对比主池 vs 备选池的表现

## 5. 风险控制

### 5.1 回退条件

如果出现以下情况，立即回退到 v0.01 baseline：

1. **EV 显著下降**：EV < v0.01 baseline * 0.8
2. **MDD 显著恶化**：MDD > v0.01 baseline * 1.5
3. **PF 崩溃**：PF < 1.5
4. **流动性问题**：超过 10% 的订单因流动性被拒绝

### 5.2 分阶段验证

每个 Phase 都必须独立验证：
1. 单独改进必须有正向效果
2. 不能"寄希望于后续改进来弥补"
3. 每个 Phase 都保留回退路径

### 5.3 证据链要求

每个改进都必须有：
1. 回测对比证据
2. 消融实验证据
3. 幂等性验证
4. 参数敏感性分析

## 8. 决策建议

### 8.1 适合选择 v0.01-improved 的场景

1. **时间紧迫**：需要在 2 个月内有可交易系统
2. **风险厌恶**：不愿意承担架构重构风险
3. **资源有限**：单人开发，精力有限
4. **渐进主义**：相信小步快跑

1. **追求理论最优**：认为前置过滤架构本质上有缺陷
2. **长期主义**：愿意投入更多时间做架构优化
